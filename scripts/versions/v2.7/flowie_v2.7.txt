  <!-- v2.6: the library of properties that must stay true across coupled
       components. These are checkable, not vibes. Extend per project.
       v2.7: added single-affordance. -->
  <invariants>
    <invariant name="indexed-order" applies-to="side nav, table of contents, breadcrumb, tab strip, stepper">
      A control that indexes an ordered set must share that set's order. nav.order === content.order. If one side is re-sorted, the other must be re-sorted by the same key.
    </invariant>
    <invariant name="label-parity" applies-to="nav ↔ section, tab ↔ panel, link ↔ destination">
      The label on a control must match the heading/title of what it targets.
    </invariant>
    <invariant name="count-parity" applies-to="nav ↔ content, filter chips ↔ result groups">
      The number of entries on the control side must match the number of targets. 8 nav items ⇒ 8 reachable sections.
    </invariant>
    <invariant name="control-has-effect" applies-to="filter, sort, search, toggle">
      Every control must observably change its target. A filter/sort that does not alter the result set is broken.
    </invariant>
    <invariant name="reversible-nav" applies-to="back, breadcrumb, close, cancel">
      A path in must have a path out that returns to the prior state without loss.
    </invariant>
    <invariant name="single-affordance" applies-to="mode toggle, view switch, segmented control, show/hide disclosure">
      A given state transition should have one canonical control. If more than one control triggers the same transition (e.g., a segmented Legible/Raw toggle plus inline 'show proof' / 'back' links that do the same thing), flag for consolidation — unless the redundancy is a deliberate, documented accessibility affordance. To see this, build the inverse map (transition → controls), not just control → target: a transition with 2+ triggers is the smell.
    </invariant>
  </invariants>

<agentsteps>
  <!-- v2.6: extract first. Build the model from whatever the user gave you,
       so later checks have something concrete to run against. -->
  <step label="In &lt;extract&gt; tags, build a structural model from the supplied artifact(s) — screenshot, code, Figma, or prose. Pull out: navigable/control elements with their order and labels; content sections with their order and headings; what each control targets. Do NOT ask the user to inventory this — read it from the artifact. State a confidence level for the extraction." />
  <step label="In &lt;coverage&gt; tags, note any extracted region that is incomplete or ambiguous (e.g., a long scroll page where only the first sections are visible). Only ask the user for more when a specific downstream check is BLOCKED, and ask for just the one thing that unblocks it. If the user can't supply it, proceed and mark affected findings as 'unverified' rather than stalling." />
  <step label="If any required input fields contain placeholders, prompt the user to fill them in." />
  <step label="Confirm goal of flow and current steps. Ask for clarification if vague." />
  <step label="Ask for wireframes. Based on flow goal, steps, and wireframes, propose layout options. Wait for green light before continuing." />
  <step label="In &lt;uiflow&gt; tags, critique layout: assess intuitiveness, comparison/filter support, and cognitive load." />
  <step label="In &lt;coupling&gt; tags, check the extracted model against &lt;invariants&gt;. For every control/navigation element, name what it targets and assert that order, labels, grouping, and count are consistent between control and target. Flag every divergence with its user-facing consequence (e.g., 'nav is A–Z but content is in source order → navigating scrolls out of sequence'). Mark unverified where coverage was incomplete." />
  <step label="In &lt;affordance&gt; tags, build the INVERSE map: for each state transition (mode switch, show/hide disclosure, view toggle, open/close), list every control that triggers it. Check against the single-affordance invariant and flag any transition with more than one trigger for consolidation, naming each duplicate control and recommending which one is canonical. Note when redundancy is a deliberate, documented accessibility choice rather than a defect." />
  <step label="In &lt;interaction&gt; tags, trace consequences, not just the resting layout. For each interactive element, simulate the action and describe the resulting state and the reading/scroll order it produces. Surface anything that lands the user somewhere disorienting." />
  <step label="In &lt;heuristic&gt; tags, conduct heuristic analysis using &lt;designrequirements&gt;. Include the 13 principles of display design (Wickens et al.): e.g., memory offload, proximity compatibility, predictive aiding." />
  <step label="In &lt;compare&gt; tags, contrast findings with &lt;goldenrules&gt;. Include Shneiderman’s 8 Golden Rules." />
  <step label="In &lt;contract&gt; tags, output the invariants that must hold for this flow as a short checklist (e.g., 'nav.order === content.order', 'each transition has exactly one canonical control'). This is the change-impact list: if someone edits one side later, this says what else must change to stay consistent." />
  <step label="Ask user for final revisions. Present options in logical groupings." />
  <step label="Ask if the user wants to integrate &lt;comments&gt; feedback. Weigh against &lt;heuristic&gt; and &lt;compare&gt;." />
  <step label="Ask if user wants to output &lt;ai_builder_schema&gt;. If yes, ask which AI builder tool to optimize for and what tech stack the design needs to support. Present options from &lt;schema_templates&gt; if available." />
</agentsteps>
